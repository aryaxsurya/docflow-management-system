Requirement Analysis & Software Requirements Specification (SRS)

Jalote menegaskan: SRS yang baik adalah fondasi seluruh proyek. "A good SRS reduces cost and effort significantly later." Tanpa ini, semua fase berikutnya menjadi tebakan.

📖 Bab 3 Jalote ⏱ 3–5 Hari 📄 Output: Dokumen SRS
1

Identifikasi Stakeholder & Elicitation Requirements

Lakukan interview atau workshop dengan calon pengguna dari ketiga peran. Gunakan teknik use case elicitation seperti yang dijelaskan Jalote di Bab 3 — ajukan pertanyaan berbasis skenario nyata.

Pertanyaan Kunci yang Harus Dijawab Siapa yang boleh melihat dokumen orang lain? Berapa maksimum level review? Apa yang terjadi jika reviewer tidak respons dalam X hari? Apakah dokumen yang di-reject bisa diajukan ulang?

Tulis Dokumen SRS (Software Requirements Specification)

Jalote di Bab 3.3 mendefinisikan SRS yang baik harus: complete, consistent, unambiguous, verifiable, dan modifiable.

  • Tujuan sistem dan ruang lingkup
  • Functional requirements per use case (dalam format: "Sistem harus ...")
  • Non-functional requirements: performance, security, usability
  • Constraint: Laravel 12, MySQL, browser-based, multi-role
  • Business rules: alur review bertingkat, aturan deadline, aturan re-submission
📄 Deliverable File SRS_DocFlow_v1.0.pdf — minimal 15 halaman, mencakup semua use case, business rules, dan batasan sistem.

Desain Arsitektur Sistem & Database Schema

Jalote Bab 5: arsitektur adalah keputusan desain paling krusial yang mempengaruhi kualitas seluruh sistem. Kesalahan di sini mahal untuk diperbaiki.

📖 Bab 5 & 6 Jalote ⏱ 3–4 Hari 📄 Output: Architecture Doc + ERD
1

Pilih & Dokumentasikan Arsitektur Aplikasi

Gunakan arsitektur MVC (Model-View-Controller) bawaan Laravel dengan beberapa tambahan layer untuk cleanness:

  • Models — Eloquent ORM untuk semua entitas data
  • Controllers — dipisah per domain: AuthController, DocumentController, ReviewController, AdminController
  • Service Layer — DocumentService, ReviewService untuk business logic kompleks (agar controller tetap tipis)
  • Form Request — validasi input dipisah dari controller
  • Policies & Gates — authorization berbasis peran
  • Events & Listeners — untuk notifikasi dan audit log (decoupled)

Implementasi Modul — Urutan Berdasarkan Dependensi

📖 Bab 6 & 7 Jalote ⏱ 15–20 Hari 🔑 Fase Terpanjang
A

Modul 1: Authentication Multi-Role & Approval Akun

Fondasi sistem. Semua modul lain bergantung pada sistem auth ini.

  • Modifikasi tabel users: tambah kolom role (enum: user, reviewer, admin) dan status (enum: pending, active, inactive)
  • Buat tiga guard atau gunakan satu guard dengan role check via middleware
  • Buat AuthController dengan login, register, logout
  • Middleware RoleMiddleware: cek role dan redirect ke dashboard yang sesuai
  • Admin dapat melihat daftar akun pending dan meng-approve atau menolak
  • Setelah register, user mendapat status pending dan tidak bisa login sampai disetujui
PHP — MIDDLEWARE
// app/Http/Middleware/RoleMiddleware.php
public function handle(Request $request, Closure $next, ...$roles): Response
{
    if (!auth()->check()) {
        return redirect('/login');
    }

    $user = auth()->user();

    if ($user->status !== 'active') {
        auth()->logout();
        return redirect('/login')->with('error', 'Akun belum disetujui admin.');
    }

    if (!in_array($user->role, $roles)) {
        abort(403, 'Akses ditolak.');
    }

    return $next($request);
}
B

Modul 2: Manajemen Dokumen (User)

CRUD dokumen lengkap dengan fitur draft, submit, dan upload lampiran.

  • Form pengajuan: judul, jenis dokumen, unit kerja, deskripsi, tanggal berlaku
  • Upload lampiran (multiple files) — simpan ke storage/app/documents/
  • Tombol "Simpan Draft" vs "Submit" — bedakan status di database
  • Autosave draft setiap 30 detik menggunakan JavaScript + AJAX endpoint
  • List dokumen milik user dengan filter status dan pencarian
  • Setelah submit, dokumen tidak bisa diedit kecuali status kembali ke draft (dari request revision)
PHP — DOCUMENT SERVICE
// app/Services/DocumentService.php
public function submit(Document $document): void
{
    if ($document->status !== 'draft') {
        throw new InvalidArgumentException('Hanya draft yang bisa disubmit.');
    }

    $document->update([
        'status'        => 'submitted',
        'current_level' => 1,
        'submitted_at'  => now(),
    ]);

    event(new DocumentSubmitted($document));
    AuditLogService::log('document.submitted', $document);
}
C

Modul 3: Alur Review Bertingkat (Core Business Logic)

Ini adalah modul paling kompleks — harus dikerjakan dengan hati-hati dan ditest secara menyeluruh.

  • Reviewer hanya bisa melihat dokumen yang current_level-nya sesuai reviewer_level mereka
  • Tiga aksi review: Approve (naik level atau selesai), Request Changes (kembali ke draft), Reject (ditolak permanen)
  • Setiap aksi disimpan ke tabel review_logs beserta catatan dan timestamp
  • Sistem otomatis tentukan apakah ada reviewer level berikutnya atau langsung ke admin
  • Monitoring deadline: hitung hari sejak dokumen masuk ke reviewer
  • Notifikasi dikirim ke reviewer saat dokumen masuk, dan ke user saat ada keputusan
PHP — REVIEW SERVICE
public function approve(Document $document, User $reviewer, string $catatan = null): void
{
    $maxLevel = User::where('role', 'reviewer')->max('reviewer_level');

    ReviewLog::create([
        'document_id'  => $document->id,
        'reviewer_id'  => $reviewer->id,
        'level'        => $document->current_level,
        'action'       => 'approved',
        'catatan'      => $catatan,
    ]);

    if ($document->current_level >= $maxLevel) {
        // Naik ke admin untuk final approval
        $document->update(['status' => 'approved']);
    } else {
        // Naik ke reviewer level berikutnya
        $document->increment('current_level');
        $document->update(['status' => 'under_review']);
        event(new ReviewLevelAdvanced($document));
    }
}

Fase 8 · Post-Launch

Maintenance, Monitoring & Iterasi Berikutnya

Jalote Bab 2.2 menyebut maintenance sebagai fase paling panjang dalam lifecycle software. Biayanya bisa mencapai 80% dari total biaya pengembangan jika tidak dikelola dengan baik.

📖 Bab 2 Jalote ⏱ Berkelanjutan
1

Setup Monitoring & Error Tracking

  • Integrasikan Sentry atau Bugsnag untuk error tracking otomatis
  • Monitor server: CPU, memory, disk usage (Netdata atau UptimeRobot)
  • Aktifkan Laravel Telescope di staging environment
  • Setup alert email jika server down atau error rate tinggi
2

Corrective & Adaptive Maintenance

Jalote membedakan tiga jenis maintenance: Corrective (perbaiki bug), Adaptive (sesuaikan dengan lingkungan baru), dan Perfective (tingkatkan performa/fitur).

  • Pantau bug report dari pengguna — respons bug kritis dalam 24 jam
  • Update dependensi secara berkala: composer update, npm update
  • Update Laravel ketika versi baru stabil (mengikuti security patches)
  • Review error log mingguan di production
3

Iterasi Fitur Berikutnya

Berdasarkan feedback pengguna, rencanakan enhancement untuk iterasi berikutnya menggunakan siklus yang sama dari Fase 1.

  • Kumpulkan feedback terstruktur dari pengguna setelah 2–4 minggu pemakaian
  • Prioritaskan enhancement berdasarkan value vs effort
  • Fitur potensial: export laporan ke PDF/Excel, integrasi email notification, mobile-responsive UI, REST API untuk integrasi sistem lain
  • Setiap enhancement mengikuti mini-cycle: requirements → design → implement → test → deploy
